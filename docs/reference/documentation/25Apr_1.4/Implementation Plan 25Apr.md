| *(Weeks 1-2)* | **1.2** | **Implement Firestore Schema Changes:** Update `applications` schema (single pass BRE fields), implement `metadata` **subcollection** structure per Data Catalog v1.4 Sec 4.11, define required indexes (`userId`, `applicationStatus`, `lastUpdatedAt`). | *(Weeks 3-5)* | **2.2** | **Implement Camunda <> Appwrite Service Call Logic:** Configure Camunda HTTP connector (or similar) within the "Get Bureau Data" Service Task to call the Appwrite API, fetch data from the 3 collections (`borrower_summary`, `enquiries`, `tradelines`) based on PAN. Handle request/response mapping. Output: Bureau data fetch orchestrated by Camunda. |
|                                      | **2.3** | **Develop Python BRE Worker Logic & Rules (Single Pass):** Set up worker project using `pyzeebe`. Implement single task handler for `calculateLoanOffer` topic. Code rules logic as detailed in Spec Sec 9.5 (Input Prep, PAN Check, Hard Rules, Risk/Calc, Final Decision including `profileFlag`). Handle I/O based on Data Catalog v1.4 (`breInputData`, `breOutputData`). Output: Python BRE worker v1 ready for deployment. |
|                                      | **2.4** | **Integrate App -> Camunda Trigger:** Implement `startLoanProcessAndFetchBureau` call in Android app (`CamundaService.kt`, `CalculateLoanOfferWithBREUseCase.kt`/ViewModel). Securely pass initial required data (e.g., `applicationId`, `userId`). Handle response (instance ID). Output: App can successfully initiate the Camunda loan workflow. |
|                                      | **2.5** | **Integrate App to Receive BRE Results:** Implement logic in Android app (ViewModel/Repository, e.g., `LoanOfferViewModel.kt`) to fetch/listen for `breOutputData` updates in the corresponding Firestore `applications` document. Parse results (`decisionStatus`, `profileFlag`, offer parameters). Output: App can receive and react to the BRE decision/offer. |
| **Phase 3: Metadata & Orchestration (Firebase Cloud Functions)** | **3.1** | **Implement Metadata Collection Points in Android App:** Integrate calls to `MetadataRepository.sendMetadataEvent` at key points (app lifecycle, screen navigation, section completion, document upload/processing events, verification events, errors) as described in Spec Sec 5.3. Ensure payload includes `eventType` and relevant data. Output: App collects metadata, stores locally in subcollection, and triggers Cloud Function endpoint. |
| *(Weeks 4-6)* | **3.2** | **Develop & Deploy Metadata Orchestrator (Cloud Functions):** Implement core logic in TypeScript (`processEvent` HTTP function, `dropOffChecker` scheduled function, Firestore interactions, `backgrounded_apps` logic) based on `cloud-functions-guide.txt`. Deploy v1 service to Firebase Functions environment. Output: Metadata Orchestrator v1 running and processing events. |
|                                      | **3.3** | **Implement App -> Orchestrator API Communication:** Define `MetadataApi.kt` Retrofit interface in App. Implement the HTTP POST call within `MetadataRepositoryImpl.sendMetadataEvent`. Ensure secure communication (HTTPS). Output: App reliably sends events to the Cloud Function endpoint. |
|                                      | **3.4** | **Implement Orchestrator Drop-off Detection Logic (Cloud Function):** Implement timer/scheduled task logic (`dropOffChecker`). Implement Firestore queries within the function to check `applications.applicationStatus`, `applications.lastUpdatedAt`, and recent `metadata` events against the `DROP_OFF_TIMEOUT_SECONDS` config. Output: Orchestrator can accurately identify potentially dropped-off applications. |
|                                      | **3.5** | **Integrate Orchestrator -> LLM Call (Drop-off Analysis - Cloud Function):** Implement LLM API call logic within `analyzeDropOff` function (TypeScript). Support configurable provider/model (Gemini 1.5 Flash / OpenAI 4o-mini). Securely handle API keys using Cloud Functions environment variables/secrets. Prepare LLM input data; parse JSON/text response. Output: Orchestrator successfully triggers LLM analysis for drop-offs. |
|                                      | **3.6** | **Integrate Orchestrator -> Email Alert (Brevo - Cloud Function):** Implement Brevo API call within `sendDropOffEmail` function (TypeScript). Format HTML email content with LLM summary. Securely handle Brevo API key using Cloud Functions environment variables/secrets. Output: Orchestrator sends formatted email alerts for detected drop-offs. |
